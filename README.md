# FlameGuard 项目概览与文档导航

FlameGuard 是一个面向“预热炉—主焚烧炉”联合系统的分层闭环控制项目。项目的总体目标是：

> 利用焚烧炉可提供的烟气余热，对高含水率厨余垃圾进行预热与脱水，并通过分层控制结构，使进入主焚烧炉的垃圾含水率维持在适合长期稳定燃烧、又能兼顾能耗与可实现性的工作区间内。

与单一静态优化器不同，FlameGuard 是一个完整的动态工程系统。它既包括垃圾组分识别、物料时延与混合、目标治理、静态运筹学优化、资源修正，也包括执行器动态、预热炉动态和焚烧炉动态响应。

本文档的作用有两个：

1. 给第一次接触项目的读者一个总览；
2. 为项目目录中的各个子模块和说明文档提供导航入口。

---

# 一、项目总体控制思路

从控制链条上看，FlameGuard 当前采用的是**分层闭环控制架构**。

其基本思想不是“直接根据当前炉温，瞬时求一个当前最优动作”，而是：

1. 先识别当前入口垃圾的组成与等效性质；
2. 再考虑这些垃圾经过前端延迟、预热炉混合、预热过程和输送延迟后，未来会以什么样的形式影响焚烧炉；
3. 外层控制器根据焚烧炉反馈与未来趋势，提出未来垃圾批次应追踪的原始目标含水率请求；
4. 目标治理模块根据当前动态阶段、稳态参考带和资源条件，把原始请求治理成当前真正允许交给内层的目标；
5. 内层静态运筹学优化器在未来槽位物料性质和资源边界已知的条件下，求出静态最优设定值；
6. 下游转译层把理想静态设定投影为执行器真正可执行的命令；
7. 执行器、预热炉和焚烧炉再通过真实动态过程形成新的反馈闭环。

因此，整个项目同时包含：

- **前馈链**：来自入口垃圾性质和烟气资源状态；
- **反馈链**：来自焚烧炉温度与烟气可测输出；
- **静态求解层**：optimizer；
- **动态对象层**：执行器、预热炉、焚烧炉。

# 二、项目模块层级

从工程功能上，当前项目可分为以下几个核心模块。

## 1. cleanser：输入清洗与等效性质映射层

职责：

- 接收原始 6 维垃圾组分比例；
- 做合法性检查与必要的归一化；
- 将原始组分映射为优化器使用的 3 个等效性质；
- 计算等效比热等派生量；
- 为 transcriber A 和 optimizer 提供标准化输入。

这是项目最上游的数据整理模块。

推荐先阅读：

- `cleanser/cleanser_readme.md`

## 2. controller：外层动态控制模块

职责：

- 读取焚烧炉可测输出；
- 通过三路反馈构造等效含水率估计；
- 结合对象动态和控制律生成原始目标请求；
- 向 governor 提供恢复方向和恢复强度。

它解决的是动态恢复问题，而不是静态设定值问题。

推荐先阅读：

- `controller/controller_readme.md`

## 3. governor：目标治理模块

职责：

- 维护稳态参考带；
- 根据当前动态状态生成动态允许带；
- 将 controller 给出的原始目标请求裁剪成最终目标；
- 向 transcriber A / optimizer 提供当前可追踪目标。

它决定“现在允许 optimizer 去追的目标是什么”。

推荐先阅读：

- `governor/governor_readme.md`

## 4. transcriber：上下游转译层

transcriber 分为两部分。

### transcriber A

职责：

- 维护未来槽位物料状态；
- 将 cleanser 输出的等效性质推进到未来出口槽位；
- 将 governor 输出的目标与未来槽位状态对齐；
- 组装成 `OptimizerRequest`。

### transcriber B

职责：

- 将 optimizer 给出的理想静态设定值投影成执行器命令；
- 做资源裁剪；
- 估算补热缺口；
- 输出最终的 `Tg_cmd`、`vg_cmd` 等执行层信号。

推荐先阅读：

- `transcriber/transcriber_readme.md`

## 5. optimizer：内层静态运筹学优化层

职责：

- 在未来槽位物料性质、目标出口含水率和资源边界已知的前提下；
- 求出静态可行、工程上合理、并按优先级最优的预热炉设定值；
- 支持三阶段词典序求解；
- 支持离线扫描制表与在线查表执行路线。

optimizer 的核心角色是：

> 未来槽位静态最优设定值生成器。

推荐先阅读：

- `optimizer/optimizer_readme.md`
- 若需要更完整建模细节，也可参考 `optimizer/optimizer_manual.md`

## 6. 动态对象层

这部分通常分散在仿真代码、测试脚本和控制层说明中，但从工程理解上必须单独认识。

### 执行器动态

当前第一版采用两个一阶环节近似：

- 风机切换时间常数约 1 s；
- 阀门切换时间常数约 3 s。

### 预热炉动态

当前第一版采用：

- 前端 5 s 纯延迟；
- 炉内垃圾混合一阶环节，时间常数约 985 s；
- 炉壁热容效应第一版可忽略或弱化。

### 焚烧炉动态

当前可测三路输出：

- 燃烧核心区域平均温度；
- 烟囱出口平均温度；
- 烟囱出口平均速度；

共享一致的动态骨架：

- 前端纯延迟约 5 s；
- 快一阶时间常数约 0.223 s；
- 慢一阶时间常数约 75.412 s；
- 仅增益系数不同。

# 四、当前项目中的关键物理量

为了方便新读者快速建立变量直觉，下面给出几个最重要的工程量。

## 1. 原始垃圾组分向量

项目最上游通常以 6 维组分比例表示厨余垃圾组成，例如：

- 菜叶；
- 西瓜皮；
- 橙子皮；
- 肉；
- 杂项混合；
- 米饭。

这些原始输入首先进入 cleanser。

## 2. 等效性质

经过 cleanser 后，当前工程主要使用三个等效性质：

- `omega0`：等效入口含水率；
- `tref`：参考干燥时间；
- `s`：温度敏感性斜率。

后续 transcriber 和 optimizer 都围绕这一组三维等效状态展开。

## 3. 外层目标量

controller 与 governor 处理的核心量是：

- `omega_req`：原始目标请求；
- `W_dyn(k)`：动态允许带；
- `omega_tar`：最终目标含水率。

## 4. 内层控制量

optimizer 求解的主要静态设定值是：

- `Tg`：预热炉入口烟气温度；
- `vg`：预热炉入口烟气速度；
- `Tm`：等效料温；
- `omega_opt`：优化器预测的出口含水率。

## 5. 焚烧炉反馈量

当前可测量的三路反馈是：

- `T_avg`：燃烧核心区域平均温度；
- `T_stack`：烟囱出口平均温度；
- `v_stack`：烟囱出口平均速度。

# 五、项目为什么采用分层而不是单一控制器

这是 FlameGuard 最重要的设计思想之一。

如果把整个问题粗暴地交给一个单一控制器，就会混在一起解决以下几件本质不同的事情：

1. 当前入口垃圾是什么；
2. 这些垃圾未来什么时候会从预热炉出来；
3. 焚烧炉当前偏热还是偏冷；
4. 当前动态阶段允许目标偏离稳态带多少；
5. 在当前资源边界下，`Tg` 和 `vg` 的静态最优设定值是什么；
6. 当前执行器究竟能不能原样执行这组设定。

这会让算法难以解释、难以调试、难以部署。

因此 FlameGuard 采用分层结构：

- **controller** 解决“恢复方向”问题；
- **governor** 解决“目标治理”问题；
- **transcriber A** 解决“时间语义对齐”问题；
- **optimizer** 解决“静态最优设定”问题；
- **transcriber B** 解决“理想解到执行命令”问题；
- **动态对象层** 解决真实物理过程传播问题。

这样的好处是：

- 每个模块职责明确；
- 参数调试更可解释；
- 后续离线制表更方便；
- 从 Python 仿真迁移到 MCU 时边界更清晰。

# 六、当前工程中几个最重要的时间尺度

FlameGuard 中最容易让新读者困惑的，是不同模块对应的时间尺度不同。当前建议记住以下几个典型量。

## 1. 测量采样频率

焚烧炉温度采样频率当前采用：

$$
10\ \mathrm{Hz}
$$

即每 0.1 s 一次。

## 2. 预热炉前端延迟

入口垃圾测量点到真正进入预热炉入口，当前第一版采用：

$$
5\ \mathrm{s}
$$

## 3. 预热炉炉内混合时间常数

由平均滞留量与进料质量流量推得：

$$
\tau_{\mathrm{mix}}=985.0\ \mathrm{s}=16.42\ \mathrm{min}
$$

## 4. 执行器动态

第一版近似为：

- 风机时间常数约 1 s；
- 阀门时间常数约 3 s。

## 5. 焚烧炉动态

当前已识别出：

- 前端延迟约 5 s；
- 快一阶约 0.223 s；
- 慢一阶约 75.412 s。

理解这些时间尺度，是理解“为什么外层、内层和对象动态必须拆开”的关键。

# 八、代码目录的理解方式

如果你打开项目目录看到很多脚本文件，建议按“模块职责”理解，而不是按“哪个文件先运行”理解。

一个典型的模块关系如下：

```text
FlameGuard/
├── cleanser/
│   ├── cleanser.py
│   └── cleanser_readme.md
├── controller/
│   ├── controller.py
│   └── controller_readme.md
├── governor/
│   └── governor_readme.md
├── transcriber/
│   ├── transcriber_a.py
│   ├── transcriber_b.py
│   └── transcriber_readme.md
├── optimizer/
│   ├── optimizer.py
│   ├── optimizer_readme.md
│   ├── optimizer_manual.md
│   └── test_optimizer.py
├── tests/
│   └── ...
└── README.md
```

这里：

- `*.py` 解决功能实现；
- `*_readme.md` 解决模块说明；
- `optimizer_manual.md` 解决更细的建模手册问题；
- `tests/` 解决仿真与验证。
